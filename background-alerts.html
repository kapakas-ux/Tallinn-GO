Tallinn GO — Background Alerts

Do alerts fire when the app is closed? Yes — and here is what to add to make them reliable.

How Local Notifications Work

✅ No background service needed.
When the user taps "Set Alert", LocalNotifications.schedule() registers the alarm directly with the Android OS alarm manager. From that moment the app can be completely closed — the OS will wake the device and show the notification at exactly the right time.
App state when alert firesWill it work?Notes
App is open / in foreground✅ YesAlways works
App is minimised (background)✅ YesAlways works
App is fully closed / swiped away✅ YesOS handles it, no JS needed
Device was rebooted after scheduling❌ LostSee "After reboot" section below
Battery saver / Doze mode active⚠️ May be lateNeeds SCHEDULE_EXACT_ALARM permission

Required: Add Missing Android Permissions

Your current AndroidManifest.xml only has INTERNET and LOCATION. You are missing two permissions that notifications need:

PermissionAndroid versionWithout it
POST_NOTIFICATIONSAndroid 13+ (API 33)Notifications silently blocked — user never sees them
SCHEDULE_EXACT_ALARMAndroid 12+ (API 31)Notification may fire minutes late due to battery optimisation
android/app/src/main/AndroidManifest.xml — add these lines inside <manifest>
<!-- Existing permissions (keep these) -->
<uses-permission android:name="android.permission.INTERNET" />
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />

<!-- Required for Android 13+: show notifications at all -->
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />

<!-- Required for Android 12+: fire at the exact scheduled minute -->
<uses-permission android:name="android.permission.SCHEDULE_EXACT_ALARM" />

<!-- Fallback for Android 13+ devices where SCHEDULE_EXACT_ALARM needs approval -->
<uses-permission android:name="android.permission.USE_EXACT_ALARM" />
⚠️ Note on USE_EXACT_ALARM: On Android 13+ Google Play may ask you to justify its use. It is allowed for alarm/reminder apps. In your Play Store description mention that the app sends departure alerts — that satisfies the review.

Optional: Reschedule After Device Reboot

If the user's phone reboots after they set an alert, Android wipes all scheduled local notifications. To fix this, add a boot receiver that re-reads the saved alerts from localStorage and reschedules them on startup.

This is done in two parts:

Part A — AndroidManifest.xml: add RECEIVE_BOOT_COMPLETED permission + receiver

<!-- Add with the other permissions -->
<uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED" />

<!-- Add inside <application> tag -->
<receiver
    android:name=".BootReceiver"
    android:exported="true"
    android:enabled="true">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED" />
        <action android:name="android.intent.action.QUICKBOOT_POWERON" />
    </intent-filter>
</receiver>

Part B — new Java file: android/app/src/main/java/…/BootReceiver.java

Place this in the same folder as MainActivity.java. Replace ee.tallinngo.app with your actual package name.

package ee.tallinngo.app;

import android.content.BroadcastReceiver;
import android.content.Context;
import android.content.Intent;
import android.content.SharedPreferences;
import android.app.AlarmManager;
import android.app.PendingIntent;
import android.util.Log;

import org.json.JSONArray;
import org.json.JSONObject;

public class BootReceiver extends BroadcastReceiver {

    private static final String TAG = "TallinnGO_Boot";

    @Override
    public void onReceive(Context context, Intent intent) {
        String action = intent.getAction();
        if (action == null) return;
        if (!action.equals(Intent.ACTION_BOOT_COMPLETED) &&
            !action.equals("android.intent.action.QUICKBOOT_POWERON")) return;

        Log.i(TAG, "Boot detected — rescheduling departure alerts");

        // Read alerts saved by the JS layer in Capacitor's WebView storage.
        // Capacitor stores localStorage in a WebView database; the easiest
        // cross-platform approach is to re-launch the activity so the JS
        // side can reschedule via LocalNotifications.schedule() itself.
        Intent launchIntent = new Intent(context, MainActivity.class);
        launchIntent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK);
        launchIntent.putExtra("reschedule_alerts", true);
        context.startActivity(launchIntent);
    }
}

Part C — src/services/alertService.ts: reschedule on app start

Add this function to alertService.ts and call it once in App.tsx on mount.

// alertService.ts — add this new export function
import { scheduleDepartureNotification } from './notificationService';

export async function rescheduleAlertsAfterBoot(): Promise<void> {
  const alerts = getActiveAlerts(); // already filters expired ones
  if (alerts.length === 0) return;

  const nowMs = Date.now();
  for (const alert of alerts) {
    const msUntilDeparture = alert.departureTimestamp - nowMs;
    const minsUntilDeparture = Math.floor(msUntilDeparture / 60000);
    if (minsUntilDeparture < alert.minutesBefore) continue; // too late
    await scheduleDepartureNotification(
      alert.stopName,
      alert.line,
      alert.destination,
      minsUntilDeparture,
      alert.minutesBefore
    );
    console.log(`Rescheduled alert: line ${alert.line} @ ${alert.stopName}`);
  }
}
// App.tsx — add inside the useEffect that already runs on mount
import { rescheduleAlertsAfterBoot } from './services/alertService';

// Inside AppContent useEffect:
rescheduleAlertsAfterBoot();

Optional: Ask User to Disable Battery Optimisation

Samsung, Xiaomi, and Huawei have aggressive battery killers that can delay even exact alarms. You can prompt the user to whitelist the app. Add this to notificationService.ts (call it once, e.g. when the first alert is set).

import { Capacitor } from '@capacitor/core';

export async function requestBatteryOptimisationExemption(): Promise<void> {
  if (!Capacitor.isNativePlatform()) return;
  try {
    // Opens the system battery optimisation settings page for this app.
    // The user can tap "Don't optimise" to ensure exact alarms fire on time.
    const { Browser } = await import('@capacitor/browser');
    await Browser.open({
      url: 'package:ee.tallinngo.app' // replace with your actual package name
    });
  } catch (e) {
    console.warn('Could not open battery settings', e);
  }
}
Simpler approach: Show a one-time dialog when the user first sets an alert, saying: "For reliable alerts, go to Settings → Battery → Tallinn GO → Don't optimise." Most users will do this if asked nicely.
